01Junior

WHAT IS A BACKGROUND JOB?

One sentence: A background job is any piece of code that runs outside the request-response life cycle.

When a client makes an API call to your server, there is a life cycle: request comes in → server processes it → server responds. Everything inside that cycle is synchronous — the client waits for the response. A background job lives outside that cycle entirely. It is triggered, but it does not make the client wait. It runs separately, in its own process, on its own timeline.

🍕 The Restaurant Analogy
A customer walks in, orders food, and the waiter takes the order — in 10 seconds the waiter confirms: "Order received!" The customer has their drink and sits comfortably. Meanwhile, in the kitchen, the chef spends 20 minutes cooking. The customer didn't wait 20 minutes at the counter. The kitchen is the background job — working hard in a separate process while the customer experience remains instant and smooth.
❌ Synchronous (Blocking)
User signs up → API waits → sends email → responds.

If the email provider is slow (3 seconds) or down, the whole signup API hangs. If it times out, the user sees an error. The signup might even fail. Bad experience.

The user's API response depends on an external service you don't control.
✅ Asynchronous (Background Job)
User signs up → API pushes task to queue → responds instantly.

Worker picks up email task and sends it independently. If the provider is down, the worker retries automatically with exponential backoff. User experience is never affected.

The user's API response depends only on your queue, which you control.
02Junior

WHY DO WE NEED THEM?

Two root problems that background jobs solve:

⏱️
Time-Consuming Work
Some operations take seconds, minutes, or hours. Encoding a video, generating a PDF report, processing a batch of 10,000 records. You cannot block an HTTP request for that long — browsers timeout after 30 seconds, load balancers timeout, users give up. These tasks must live outside the request cycle.
🔗
External Service Dependencies
Any time your server calls another server (email provider, SMS provider, push notification service, payment gateway), you introduce failure risk you cannot control. External services go down. They have rate limits. They are slow sometimes. Offloading these calls to background jobs with retry logic protects your users from those failures.
💡 The Two Rules of Thumb
Rule 1: If a task takes more than ~200ms OR depends on an external service → consider making it a background job.
Rule 2: If the result of a task does not need to be in the HTTP response → it should be a background job.
03Junior

SYNC VS ASYNC PROCESSING — The Email Example

Let's walk through exactly what goes wrong with synchronous processing, and exactly how background jobs fix it, using the signup verification email as the example.

// ❌ SYNCHRONOUS — everything in one request
1
User submits signup form
Frontend makes POST /auth/signup. User is waiting. Clock starts ticking.
2
Server validates, hashes password, saves to DB
Fast. Takes ~50ms. User still waiting.
3
Server calls email provider API (Resend / Mailgun)
Making an HTTP call to an external server. Takes 200ms–3000ms normally. User still waiting. If the provider is DOWN — this call hangs until timeout (30s+). Signup appears frozen. User gives up.
4
Email provider returns error → your signup API fails
You either return 500 (user thinks signup failed — they try again, creating duplicate) OR you swallow the error and tell user email was sent — it wasn't. Both are terrible outcomes.
// ✅ ASYNCHRONOUS — background job pattern
1
User submits signup form
Frontend makes POST /auth/signup. User is waiting.
2
Server validates, saves to DB, generates verification code
Same as before. ~50ms.
3
Server serializes task data and PUSHES to queue
Packages { userId, userEmail, verificationCode, template } as JSON and pushes to the email queue. This takes ~2ms. Queue is YOUR infrastructure — always fast.
4
Server returns 201 — INSTANTLY
Total API response time: ~55ms. User sees "Check your email for verification." User experience is perfect regardless of what happens next.
5
Worker picks up task, calls email provider
In a SEPARATE PROCESS. Worker dequeues the task, deserializes it, calls Resend/Mailgun API. If it fails → retry with exponential backoff. Email provider comes back up → succeeds automatically. User gets their email. Nobody had to do anything.

WHEN TO OFFLOAD TO BACKGROUND

📧
Sending Emails
Verification, welcome, password reset, notification emails. Always external provider — always background. Resend, Mailgun, SendGrid all have occasional downtime.
🖼️
Image / Video Processing
Resizing images to multiple breakpoints (mobile, tablet, desktop). Encoding video to multiple resolutions (480p, 720p, 1080p). These can take seconds to minutes.
📄
Report Generation
Generating PDF weekly reports, pulling analytics data, constructing HTML email summaries. Can involve complex DB queries across millions of rows.
🔔
Push Notifications
Mobile push via APNs (Apple) or FCM (Google). Your backend calls their external APIs. Must be offloaded — external service call, can fail, needs retry.
🗑️
Account Deletion
Deleting all user data across tables, S3 assets, related records. Can take minutes. Cannot block an API request. Offload and respond immediately.
🔄
DB Cleanup / Maintenance
Deleting expired sessions, cleaning orphan records, archiving old data. Run on a schedule — nightly or weekly — never blocking user requests.
🤖
AI / ML Processing
Running inference on uploaded files (image classification, transcription, sentiment analysis). Often slow, GPU-bound — always a background job.
💳
Webhooks / External Events
Processing incoming webhooks from Stripe, GitHub, Twilio. Acknowledge immediately (200 OK), process in background. External systems have timeout expectations (~30s).

PRODUCER → QUEUE → CONSUMER

The architecture behind every background job system is the Producer-Consumer pattern. Three components, three clear responsibilities.

// the three-component architecture
🏭 PRODUCER
Your API / App Code
Creates & enqueues tasks
──────►
📦 BROKER / QUEUE
Redis / RabbitMQ / SQS
Stores tasks until consumed
──────►
⚙️ CONSUMER / WORKER
Separate Process
Dequeues & executes tasks
Worker sends ACK ✓
──────►
Queue removes task
Worker fails ✗
──────►
Queue retries task
🏭 Producer
What it is: Your main application code — the Express/Fastify/NestJS server.

What it does: Creates a task object with all the data the worker will need, serializes it to JSON, and pushes it onto a named queue.

What it does NOT do: Execute the task. Once it's enqueued, the producer's job is done.
📦 Broker (Queue)
What it is: A message store — Redis, RabbitMQ, Amazon SQS, etc.

What it does: Stores tasks durably until a worker is ready. Tracks task state (waiting, active, completed, failed). Manages retries. Ensures no task is lost.

The "to-do list" for your backend.
⚙️ Consumer (Worker)
What it is: A separate process — runs independently of your main API server.

What it does: Constantly polls the queue. Picks up tasks. Deserializes JSON back to native objects. Runs the handler function. Sends acknowledgement back.

Can be scaled horizontally by adding more worker instances.

ENQUEUE & DEQUEUE

Enqueue = adding a task to the queue (producer's job). Dequeue = taking a task out of the queue (consumer's job). These are the two fundamental queue operations — the same concepts you know from data structures.

Enqueue & Dequeue — Conceptual Flow
// ─── ENQUEUE (Producer side) ────────────────────────────────────── // When user signs up, producer creates a task and pushes to queue const taskData = { userId: 'user_abc123', email: 'alice@example.com', name: 'Alice', code: '847392', // verification code template: 'email-verification' }; // Serialize to JSON → push to 'email' queue // This takes ~2ms. API responds immediately after this. await emailQueue.add('send-verification', taskData); // ↑ "send-verification" is the JOB NAME (type of task) // taskData is serialized to JSON and stored in Redis // ─── DEQUEUE (Consumer side — runs in SEPARATE PROCESS) ─────────── // Worker constantly monitors the queue // When a new task arrives, it automatically dequeues it emailQueue.process('send-verification', async (job) => { // job.data is the deserialized task object const { userId, email, name, code, template } = job.data; // Run the actual work await sendEmail({ to: email, from: 'noreply@yourapp.com', subject: 'Verify your account', html: renderTemplate(template, { name, code }) }); // If we reach here without throwing → worker sends ACK automatically // Queue marks this task as COMPLETED and removes it // If sendEmail() throws → worker sends NACK automatically // Queue marks task as FAILED and schedules a retry }); // ─── QUEUE STATE at any moment ──────────────────────────────────── // waiting → tasks queued, waiting for a free worker // active → task dequeued, worker is processing it right now // completed → task finished successfully, ACK received // failed → task threw an error, will be retried // delayed → task scheduled for future, not yet in waiting
08Mid-Level

ACKNOWLEDGEMENTS — How the Queue Knows

When a worker finishes a task, it sends an acknowledgement (ACK) back to the queue. This tells the queue: "Task completed successfully. Remove it." If the worker does NOT send an ACK — because it crashed, threw an error, or the process was killed — the queue knows something went wrong and can take action.

✅ ACK — Success
Worker completes the handler function without throwing an error. Framework automatically sends ACK to the queue. Queue removes the task from storage. Task appears in "completed" state.
❌ NACK — Failure
Worker throws an error (or crashes). Framework sends NACK (negative acknowledgement). Queue marks task as "failed" and schedules it for retry according to the backoff configuration.
💡 Why Acknowledgements Matter
Without acknowledgements, you'd have no way to know if a task succeeded or failed. A task dequeued by a crashing worker would simply disappear — lost forever. The ACK/NACK system is what makes task queues reliable. Every task is guaranteed to be processed at least once (and usually exactly once with idempotency design).
09Mid-Level

VISIBILITY TIMEOUT

The visibility timeout is the maximum time the queue gives a worker to finish processing a task and send an acknowledgement. If the worker doesn't ACK within this window, the queue assumes the worker crashed or hung — and makes the task available to other workers.

📬 The Delivery Analogy
A post office hands a package to a courier. The courier has 30 minutes to deliver it and confirm. If the courier doesn't confirm in 30 minutes — maybe their vehicle broke down — the post office takes the package back and gives it to a different courier. The package is never lost. That 30-minute window is the visibility timeout.
Visibility Timeout — Configuration & Behavior
// ─── BullMQ: lock duration = visibility timeout ─────────────────── const worker = new Worker('email', async (job) => { await sendEmail(job.data); }, { lockDuration: 30000, // 30 seconds — worker must finish within this window // If it doesn't, queue makes task available to other workers stalledInterval: 5000, // Check for stalled workers every 5 seconds }); // ─── Amazon SQS: VisibilityTimeout parameter ────────────────────── const message = await sqs.receiveMessage({ QueueUrl: process.env.QUEUE_URL, VisibilityTimeout: 60, // 60 seconds — SQS hides message from other consumers MaxNumberOfMessages: 1, }).promise(); // Worker has 60 seconds to process and delete the message // If it doesn't delete within 60s, SQS makes it visible again // ─── Choosing the right timeout ─────────────────────────────────── // Simple email task: 30 seconds (usually completes in <5s) // Image processing: 5 minutes (can take 30-90 seconds) // Video encoding: 30 minutes (can take minutes) // PDF report generation: 2 minutes (DB queries + PDF build) // Rule: set to 2-3x the expected max execution time // ─── Heartbeat for long-running tasks ──────────────────────────── // For tasks that take longer than the timeout, extend the lock periodically const worker = new Worker('video-encode', async (job) => { const heartbeat = setInterval(() => job.updateProgress(50), 10000); try { await encodeVideo(job.data); } finally { clearInterval(heartbeat); } }, { lockDuration: 300000 }); // 5 minute timeout for video tasks

TYPE 3 — Chain Tasks

Chain tasks have a parent-child relationship. Task B cannot start until Task A completes. Some tasks can run in parallel if they don't depend on each other. The classic example is video upload processing on an LMS (Udemy-style) platform.

① Video Upload
User uploads → S3 via presigned URL → triggers chain
→
② Video Encoding
Encode to 480p, 720p, 1080p — PARENT TASK
All subsequent tasks depend on this
③a Thumbnail Generation
Extract frames → generate thumbnails
④ Thumbnail Processing
Resize to multiple breakpoints
Depends on ③a
parallel
③b Transcription
Audio → subtitles / captions
Runs in parallel with ③a
💡 Key Rules of Chain Tasks
1. A child task only starts when its parent task succeeds. If video encoding fails, thumbnail generation never starts — there's nothing to generate from.

2. Independent children run in parallel. Thumbnail generation and transcription both depend on encoding, but not on each other — run them simultaneously to save time.

3. Failure in a child does NOT fail the parent. If transcription fails, the video is still uploaded and encoded. The retry mechanism handles transcription separately.
BullMQ — Chain Tasks with Flow
// BullMQ has a FlowProducer for chaining tasks import { FlowProducer } from 'bullmq'; const flowProducer = new FlowProducer({ connection: redisConfig }); // Define the chain: parent + children // Children only run after the parent completes await flowProducer.add({ name: 'encode-video', // PARENT queueName: 'video-processing', data: { videoId: 'vid_abc123', s3Key: 'uploads/vid_abc123.mp4' }, children: [ { name: 'generate-thumbnails', // CHILD 1 — runs after parent queueName: 'thumbnail', data: { videoId: 'vid_abc123' }, children: [ { name: 'process-thumbnails', // GRANDCHILD — runs after child 1 queueName: 'image-processing', data: { videoId: 'vid_abc123' } } ] }, { name: 'generate-transcription', // CHILD 2 — runs in parallel with child 1 queueName: 'transcription', data: { videoId: 'vid_abc123' } } ] });
13Mid-Level

TYPE 4 — Batch Tasks

Batch tasks involve processing a large number of items, usually by spawning many individual tasks from a single trigger. Two patterns: one task spawns many sub-tasks, or a scheduler fires many tasks simultaneously.

Example 1: Account Deletion
User clicks "Delete Account". API responds instantly with 200.

In background, the DELETE_ACCOUNT task spawns sub-tasks:
→ delete-user-projects
→ delete-user-assets (S3 files)
→ delete-user-profile
→ send-goodbye-email
→ delete-user-record

Each sub-task is small, focused, and independently retriable. If deleting assets fails, it retries without restarting the whole deletion.
Example 2: Weekly Report Send
Cron fires every Sunday at midnight.

It queries all users with reports enabled.
For each user, it enqueues a GENERATE_AND_SEND_REPORT task.

If you have 50,000 users → 50,000 tasks are enqueued simultaneously. Workers process them in parallel. All reports delivered within minutes.

Without background jobs, you couldn't do this at all — you'd need to run a single synchronous loop for hours.
Batch Task — Spawning Sub-Tasks from a Parent
// ─── Handler: DELETE_ACCOUNT task spawns sub-tasks ───────────────── deleteAccountWorker.process('delete-account', async (job) => { const { userId } = job.data; // Get all resources this user owns const [projects, assets] = await Promise.all([ db.query('SELECT id FROM projects WHERE owner_id = $1', [userId]), db.query('SELECT s3_key FROM user_assets WHERE user_id = $1', [userId]), ]); // Spawn sub-tasks in BULK (efficient — single Redis command) await deletionQueue.addBulk([ ...projects.rows.map(p => ({ name: 'delete-project', data: { projectId: p.id, userId } })), ...assets.rows.map(a => ({ name: 'delete-s3-asset', data: { s3Key: a.s3_key } })), { name: 'delete-user-profile', data: { userId } }, { name: 'send-goodbye-email', data: { userId } }, ]); // All sub-tasks run in parallel, each independently retriable }); // ─── Weekly Reports — Batch Enqueue ────────────────────────────── reportCronWorker.process('weekly-reports-trigger', async () => { const users = await db.query( 'SELECT id, email FROM users WHERE weekly_report_enabled = true' ); // Enqueue ALL user report tasks at once — bulk is much faster than loop await reportQueue.addBulk( users.rows.map(user => ({ name: 'generate-user-report', data: { userId: user.id, email: user.email }, opts: { attempts: 3, backoff: { type: 'exponential', delay: 60000 } } })) ); // 50,000 tasks enqueued → workers process in parallel → all done in minutes });
14Mid-Level

RETRIES — Exponential Backoff

When a task fails, you don't want to immediately retry it. If the external service is down, retrying in 1 second will fail again. Exponential backoff means increasing the delay between retries exponentially — giving the external service time to recover while not hammering it with retries.

// ATTEMPT 1
FAIL
email API down
// RETRY 1
wait 1 min
still down
// RETRY 2
wait 2 min
still down
// RETRY 3
wait 4 min
back up ✓
// RESULT
✓
email sent
Exponential Backoff — BullMQ Configuration
// ─── Job Options with Retry Configuration ──────────────────────── await emailQueue.add('send-verification', taskData, { attempts: 5, // max 5 total attempts (1 original + 4 retries) backoff: { type: 'exponential', // delay doubles on each failure delay: 60000, // start with 60 seconds (1 min) }, // Retry schedule: // Attempt 1: immediate // Retry 1: 60s (1 min) // Retry 2: 120s (2 min) // Retry 3: 240s (4 min) // Retry 4: 480s (8 min) // After 5th failure → moves to Dead Letter Queue removeOnComplete: { count: 1000 }, // keep last 1000 completed jobs for inspection removeOnFail: { count: 5000 }, // keep last 5000 failed jobs in DLQ }); // ─── Different configs for different task types ─────────────────── const JOB_OPTIONS = { // Email: up to 5 retries, exponential starting 1 min email: { attempts: 5, backoff: { type: 'exponential', delay: 60_000 } }, // Image processing: 3 retries, linear 30s backoff imageProcess: { attempts: 3, backoff: { type: 'fixed', delay: 30_000 } }, // Payment: up to 3 retries over 24 hours payment: { attempts: 3, backoff: { type: 'exponential', delay: 3_600_000 } // 1h, 2h, 4h }, // Non-critical notification: 2 retries, fail fast notification: { attempts: 2, backoff: { type: 'fixed', delay: 5_000 } } }; // ─── Access retry info inside the handler ───────────────────────── emailWorker.process(async (job) => { console.log(`Attempt ${job.attemptsMade + 1} of ${job.opts.attempts}`); try { await sendEmail(job.data); } catch (err) { // Log every failure with context console.error({ jobId: job.id, attempt: job.attemptsMade + 1, error: err.message, data: job.data, }); throw err; // Re-throw to trigger retry mechanism } });
15Senior

IDEMPOTENCY — The Most Critical Design Principle

Idempotency means: running a task multiple times produces the same result as running it once. This is critical because retries are fundamental to background job systems. If your email task fails and retries, you do NOT want the user to receive two emails. If your delete-account task retries halfway through, you do NOT want it to crash because it tries to delete something already deleted.

💡 The Light Switch Analogy
A light switch is idempotent. Pressing "ON" when the light is already on does nothing harmful. But a counter "increment by 1 on every click" is NOT idempotent — clicking it multiple times changes the result. Your tasks must be light switches, not counters — safe to execute multiple times with the same outcome.
Idempotency Patterns — How to Design Safe Tasks
// ─── PROBLEM: Non-Idempotent Email Task ─────────────────────────── // If this task retries, the user gets TWO emails. Bad. emailWorker.process(async (job) => { await sendEmail(job.data); // ← retries = duplicate emails ❌ }); // ─── SOLUTION 1: Idempotency Key ───────────────────────────────── // Before sending, check if we already sent this specific email emailWorker.process(async (job) => { const idempotencyKey = `email_sent:${job.id}`; // Check if already sent (Redis SET NX = atomic "set if not exists") const alreadySent = await redis.set(idempotencyKey, '1', 'NX', 'EX', 86400); if (!alreadySent) { console.log('Email already sent, skipping duplicate'); return; // ACK without doing anything — safe to skip } await sendEmail(job.data); // Only runs if key was NOT already set }); // ─── SOLUTION 2: Database Status Check ──────────────────────────── // Track in DB whether the operation already happened accountDeletionWorker.process('delete-project', async (job) => { const { projectId, userId } = job.data; // Check if project is already deleted (idempotency check) const project = await db.query( 'SELECT id, deleted_at FROM projects WHERE id = $1', [projectId] ); if (!project.rows[0] || project.rows[0].deleted_at) { return; // Already deleted — safe to return without error } // Use a DB TRANSACTION — all-or-nothing const client = await db.connect(); try { await client.query('BEGIN'); await client.query('DELETE FROM project_members WHERE project_id = $1', [projectId]); await client.query('DELETE FROM tasks WHERE project_id = $1', [projectId]); await client.query('DELETE FROM projects WHERE id = $1', [projectId]); await client.query('COMMIT'); } catch (err) { await client.query('ROLLBACK'); throw err; // Re-throw → triggers retry from scratch (transaction rolled back = clean state) } finally { client.release(); } }); // ─── SOLUTION 3: Upsert instead of Insert ──────────────────────── // If your task creates a record, use INSERT ... ON CONFLICT DO NOTHING await db.query(` INSERT INTO email_logs (job_id, user_id, email_type, sent_at) VALUES ($1, $2, $3, NOW()) ON CONFLICT (job_id) DO NOTHING `, [job.id, userId, emailType]); // If this runs twice (retry), the second insert is silently ignored
16Mid-Level

DEAD LETTER QUEUE (DLQ)

When a task exhausts all its retry attempts and still fails, it moves to the Dead Letter Queue. The DLQ is a separate queue that holds permanently-failed tasks. You don't lose them — you store them safely so you can investigate, fix the bug, and re-queue them manually.

⚠️ Without a DLQ
If you have no DLQ and a task fails all retries, the task data is gone forever. You have no way to know which users didn't receive their email, which accounts weren't deleted, which reports weren't generated. Production systems MUST have a DLQ and an alerting system that fires when tasks land in it.
Dead Letter Queue — Setup & Monitoring
// ─── BullMQ: listen for failed events ──────────────────────────── const queueEvents = new QueueEvents('email', { connection: redisConfig }); queueEvents.on('failed', async ({ jobId, failedReason }) => { const job = await emailQueue.getJob(jobId); if (job && job.attemptsMade >= job.opts.attempts) { // This job has exhausted ALL retries → it's in the DLQ console.error('JOB PERMANENTLY FAILED', { jobId, failedReason, data: job.data, }); // 1. Store in database for investigation await db.query(` INSERT INTO failed_jobs (job_id, queue, data, error, failed_at) VALUES ($1, $2, $3, $4, NOW()) `, [jobId, 'email', JSON.stringify(job.data), failedReason]); // 2. Send alert to engineering team (Slack, PagerDuty, email) await sendSlackAlert({ channel: '#alerts-background-jobs', text: `🚨 Job permanently failed: ${job.name} | ${failedReason}`, data: job.data, }); } }); // ─── Manual retry from DLQ (after fixing the bug) ──────────────── async function retryDeadLetterJobs(queueName: string) { const queue = new Queue(queueName, { connection: redisConfig }); const failedJobs = await queue.getFailed(); for (const job of failedJobs) { await job.retry(); // Re-queue the job after fixing the underlying issue } console.log(`Retried ${failedJobs.length} failed jobs from ${queueName}`); }

⚠️ Cron in a Distributed System
If you have 3 worker instances and a cron fires, ALL 3 workers will try to execute it — causing the task to run 3 times. Prevent this by using a distributed lock (Redis SET NX at the start of the cron handler) OR by using BullMQ's built-in scheduler (which ensures only one instance picks up the repeatable job). Never run cron on multiple instances without a lock.

THE PRODUCTION DESIGN FRAMEWORK

When a client gives you a project and asks you to design the background job system, here is the 8-step framework to think through it systematically. This is what senior engineers do — they think before they code.

01
Identify Every Background Task
Go through the entire feature list and ask: "Does this need to happen in the request-response cycle?" For anything involving external services, heavy computation, or operations that can take more than 200ms — mark it as a background job. Create a list: emails, notifications, media processing, report generation, data sync, webhooks, scheduled cleanup.
02
Classify Each Task by Type
Label each task: One-Off (triggered by a user action), Recurring (scheduled on a cron), Chain (depends on another task), or Batch (spawns multiple sub-tasks). This classification tells you how to configure the queue — chains need FlowProducer, crons need repeat patterns, batches need addBulk.
03
Design Queue Architecture — Separate Queues Per Domain
Never put all jobs in one queue. Create separate queues for each concern: email, media-processing, notifications, reports, scheduler, deletions. This lets you scale workers independently (add more email workers during a campaign), set different concurrency per queue, and monitor each domain separately. One slow media queue won't block email delivery.
04
Configure Retry Strategy Per Task Type
Not all tasks need the same retry behavior. Emails: 5 retries, exponential from 1min. Payments: 3 retries, 1h/2h/4h. Image processing: 3 retries, linear 30s. Notifications: 2 retries, 5s. Set appropriate visibility timeouts too — media processing needs minutes, email needs seconds.
05
Design for Idempotency From the Start
For every task handler, ask: "If this runs twice, what breaks?" Then fix it. Use idempotency keys in Redis (SET NX) for external calls. Use INSERT ... ON CONFLICT DO NOTHING for DB records. Use transactions with rollback for multi-step DB operations. Assume every task WILL run at least twice — design accordingly.
06
Set Up Dead Letter Queue + Alerting
Every queue must have a DLQ strategy. When jobs exhaust retries: store them in a failed_jobs DB table, send an alert to Slack/PagerDuty, and make them manually retriable after you fix the bug. Without this, permanently failed jobs disappear silently. In production, DLQ alerts should wake someone up.
07
Plan Worker Scaling Strategy
Workers must be stateless — any worker can handle any task. Set concurrency per worker (how many jobs one worker processes simultaneously). Plan for horizontal scaling — how many worker instances will you run? In Kubernetes, workers are separate Deployments from your API. Scale workers independently based on queue depth metrics.
08
Set Up Monitoring Before Launch
Track: queue depth (waiting jobs), processing rate (jobs/minute), error rate (failed %), worker health (uptime). Set alerts: if queue depth > 1000 → scale workers. If error rate > 5% → investigate. If any worker is down → alert immediately. Use BullBoard for a real-time dashboard. Emit metrics to Prometheus/Grafana for historical analysis.

BEST PRACTICES

📦
Keep Tasks Small & Focused
One task = one unit of work. Don't send email AND process image AND update DB in one task. If one thing fails, the whole task retries — wasting work already done. Split into separate tasks, use chains for dependencies.
⚡
Avoid Long-Running Tasks
Tasks that take more than a few minutes are hard to monitor, hard to retry safely, and create visibility timeout issues. Break them into smaller chunks using chain tasks or batch tasks.
🔁
Design for Idempotency
Every task WILL be retried. Design every handler to be safely re-executable. Idempotency keys, ON CONFLICT DO NOTHING, status checks. This is non-negotiable in production.
📋
Structured Logging
Log at start and end of every task. Log every failure with full context (jobId, taskData, error, attempt number). Use JSON logs for searchability. Without good logs, debugging background failures is nearly impossible.
🚨
Never Swallow Errors
Always re-throw errors in task handlers. Swallowing errors prevents retries — the queue thinks the task succeeded. If an error happens, throw it. Let the queue handle retrying.
🔒
Distributed Lock for Crons
If running multiple worker instances, use Redis SET NX at the start of every cron handler. Without it, all worker instances will execute the cron simultaneously — sending duplicate reports to thousands of users.
📊
Monitor Queue Depth
Set alerts when queue depth exceeds a threshold. Growing queue depth = workers can't keep up = tasks getting delayed. Respond by scaling workers horizontally. Never let the queue grow unbounded.
🔄
Graceful Shutdown
On SIGTERM, tell workers to stop accepting new jobs but finish current ones. Never kill a worker process mid-task — you'll have partially-completed tasks and inconsistent data. BullMQ's worker.close() handles this.
